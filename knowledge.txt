题目: 进程与线程的区别
关键点: 资源分配, 调度, 并发, 切换开销
描述: 进程是资源分配的基本单位，线程是调度的基本单位。

题目: Java 三大特性（封装、继承、多态）
关键点: 封装，继承，多态，重载，重写
描述: 封装通过private修饰成员变量+public getter/setter控制访问；继承用extends关键字，子类继承父类非私有成员，final类不可继承；多态通过方法重写（Override）与接口实现实现。重载是方法名相同参数列表不同（编译期），重写是子类覆盖父类方法（运行期）。

题目: equals 与 == 的区别
关键点: 内存地址比较，内容比较，重写equals，hashCode
描述: == 比较的是栈内存中存放的对象内存地址（基本类型比值，引用类型比地址）；equals 比较两个对象的内容是否相等。Object类默认equals就是用==，需根据业务在自定义类中重写equals方法，通常还要重写hashCode。

题目: final 与 static 的用法
关键点: final类，final方法，final变量，static变量，static方法，静态代码块
描述: final修饰类则不可继承（如String）；修饰方法则不可重写；修饰变量则不可改变（基本类型值不变，引用类型引用不变但对象内部可改）。static修饰的属性和方法属于类级别，可通过类名直接调用，静态代码块在类加载时执行一次。

题目: ArrayList 与 LinkedList 的区别
关键点: 动态数组，双向链表，随机访问，插入删除，时间复杂度
描述: ArrayList基于动态数组，支持随机访问get(i)时间O(1)，插入/删除中间元素需移动后续元素O(n)，适用于频繁查询场景；LinkedList基于双向链表，插入/删除头尾元素O(1)，随机访问需遍历O(n)，适用于频繁插入删除场景。ArrayList初始容量10，扩容为1.5倍。

题目: HashMap 的底层实现与扩容机制
关键点: 数组+链表+红黑树，哈希冲突，负载因子，扩容
描述: HashMap底层为数组+链表/红黑树，哈希冲突通过链地址法解决。初始容量16，负载因子0.75，扩容时容量翻倍并重新哈希（rehash）。

题目: ConcurrentHashMap 的线程安全实现
关键点: CAS，同步锁，分段锁，JDK8
描述: JDK8后ConcurrentHashMap采用CAS（Compare-And-Swap）+同步锁优化并发性能，相比JDK7的分段锁机制，锁粒度更细，并发度更高。

题目: 实现线程安全的三种方式
关键点: synchronized，Lock接口，原子类，CAS，volatile
描述: ①同步代码块：使用synchronized关键字锁定对象；②Lock接口：提供更灵活锁操作（如可中断锁、公平锁），如ReentrantLock；③原子类：利用CAS实现无锁并发，如AtomicInteger。volatile保证可见性但不保证原子性。

题目: 进程与线程的区别
关键点: 资源分配，调度，并发，切换开销
描述: 进程是资源分配的基本单位，线程是调度的基本单位。进程间切换开销大，线程间切换开销小；进程间资源独立，线程共享进程资源。

题目: JVM 运行时数据区域
关键点: 程序计数器，虚拟机栈，本地方法栈，堆，方法区
描述: JVM运行时数据区主要分为：程序计数器（当前线程字节码行号指示器）、Java虚拟机栈（线程私有，每个方法创建栈帧）、本地方法栈、堆（所有线程共享，存储对象实例）、方法区（存储类信息、常量、静态变量）。

题目: 垃圾回收算法
关键点: 标记-清除，复制算法，标记-整理，分代收集
描述: 主要垃圾回收算法包括：标记-清除（有碎片）、复制算法（无碎片但浪费内存）、标记-整理（无碎片，适合老年代）。HotSpot虚拟机采用分代收集，新生代用复制算法，老年代用标记-整理或标记-清除。

题目: Spring 框架的核心功能（IoC 和 AOP）
关键点: 控制反转，依赖注入，面向切面编程，Bean生命周期
描述: Spring的核心是IoC（控制反转）容器，负责管理对象的创建、配置和生命周期，通过DI（依赖注入）实现解耦。AOP（面向切面编程）将横切关注点（如日志、事务）从业务逻辑中分离，通过动态代理实现。

题目: Spring Boot 自动配置原理

题目: MySQL 索引类型与失效场景
关键点: B+Tree索引，哈希索引，复合索引，最左前缀原则
描述: MySQL主要索引类型包括B+Tree索引（InnoDB默认）、哈希索引等。索引失效的常见场景：①违反最左前缀原则；②在索引列上进行函数运算或类型转换；③使用LIKE以%开头；④使用OR连接非索引列。

题目: MySQL 事务隔离级别
关键点: 读未提交，读已提交，可重复读，串行化，脏读，不可重复读，幻读
描述: InnoDB支持四种事务隔离级别：①读未提交（READ UNCOMMITTED）：最低级别，会出现脏读；②读已提交（READ COMMITTED）：避免脏读，但会出现不可重复读；③可重复读（REPEATABLE READ）：MySQL默认级别，避免脏读和不可重复读，但可能出现幻读；④串行化（SERIALIZABLE）：最高级别，完全隔离但性能最低。

题目: AI Agent 的定义与核心组件
关键点: 大语言模型，感知环境，自主决策，工具调用，记忆模块
描述: AI Agent是基于大语言模型构建的智能体，具备感知环境、自主决策和执行任务的能力。核心组件包括：①感知层（接收输入/环境信号）；②决策层（规划模块分解任务、记忆模块存储上下文）；③执行层（调用工具/API完成任务）。

题目: Agent 与传统 AI 聊天机器人的核心区别
关键点: 动态规划，多步任务，工具调用，目标导向
描述: Agent具备动态规划多步任务的能力（如订票+推荐），可调用外部工具获取实时信息，以长期目标驱动行动。传统聊天机器人仅做单轮问答，依赖预训练知识库，为短期请求响应。

题目: ReAct 框架的原理
关键点: Reasoning，Acting，思维链，思考-行动循环
描述: ReAct = Reasoning + Acting，将“推理”（思考下一步该做什么）和“行动”（执行具体操作）交替进行。Agent先“思考”当前状态和目标，再“行动”调用工具或给出回答，通过“思考-行动-观察”的循环完成复杂任务。

题目: Memory 系统的设计（短期记忆与长期记忆）
关键点: 短期记忆，长期记忆，向量数据库，知识库
描述: Agent的短期记忆存储当前会话的上下文（类似工作记忆），长期记忆存储历史经验和知识（可借助向量数据库或知识库）。设计时需考虑记忆的存储、检索、更新和遗忘机制。

题目: Tool Use（工具调用）的实现方式
关键点: Function Calling，API调用，工具描述，错误处理
描述: 工具调用指Agent通过API或函数调用与外部系统交互。技术要点：①工具描述（定义清晰的工具接口，如OpenAPI规范）；②调用策略（LLM生成格式化的工具调用指令，如JSON）；③错误处理（重试机制和fallback方案）。

题目: LangChain 与 LlamaIndex 框架的对比
关键点: 框架选型，Agent编排，数据索引，应用场景
描述: LangChain侧重Agent编排和链式调用，适合构建复杂的多步骤Agent工作流；LlamaIndex侧重数据索引和检索，适合构建RAG系统。选型时需根据核心需求（是复杂Agent逻辑还是数据检索）决定。

题目: 多智能体系统（Multi-Agent）的优势与挑战
关键点: 多Agent协作，任务分解，通信协调，系统复杂性
描述: 多Agent系统的优势包括任务分解并行处理、角色分工专业化。引入的复杂性包括：Agent间通信与协调成本增加、任务分配策略设计困难、一致性和冲突解决挑战。

题目: 限制 Agent 能力和普及的最大瓶颈
关键点: 模型能力，成本，可靠性，安全性
描述: 主要瓶颈包括：①大模型本身的推理和规划能力仍有限；②API调用成本和延迟较高；③Agent行为的安全性和可控性难以保障；④长期记忆和持续学习机制尚不成熟。

题目: 评估 Agent 为何比评估基础 LLM 更困难
关键点: 多步骤任务，环境交互，目标达成度，基准测试
描述: 评估Agent比评估基础LLM更复杂，因为Agent需要完成多步骤任务、与环境交互，评估维度包括任务完成率、执行效率、工具调用准确性、决策合理性等。专门的Agent评估基准需要构建复杂的测试环境和任务。

题目: RAG 的工作原理
关键点: 检索，增强，生成，外部知识库，向量检索
描述: RAG（Retrieval-Augmented Generation）是一种结合“外部知识检索”和“大语言模型生成”的混合架构。工作流程分三步：①基于用户查询从知识库中检索相关文档；②通过向量相似度计算实现粗排；③将检索到的文档作为上下文输入LLM生成回答。

题目: RAG 与模型微调（SFT）的区别
关键点: 知识注入方式，更新成本，幻觉，可溯源
描述: SFT是在模型内部“灌知识”（靠记忆），RAG是让模型“查资料”（靠检索）。RAG的优势：知识更新快、成本低、回答可溯源；SFT的优势：推理更自然流畅。RAG主要解决大模型知识过时、缺少私有数据、回答不可溯源等问题。

题目: RAG 的核心优势
关键点: 降低幻觉，知识实时更新，成本低，可溯源
描述: RAG的核心优势包括：①降低幻觉（Hallucination），让回答更贴近事实；②节省训练成本，无需让模型“死记硬背”海量数据；③知识可插拔，更新方便；④回答可溯源，增强可信度。

题目: RAG 完整流水线的关键步骤
关键点: 文档处理，文本切块，向量化，索引构建，检索，生成
描述: 完整RAG流水线包括：①数据准备（文档解析、清洗）；②文本切块（Chunking）；③向量化（Embedding）；④索引构建（存入向量数据库）；⑤检索（根据用户查询召回相关块）；⑥生成（将检索结果作为上下文输入LLM）。

题目: 文本切块策略的设计权衡
关键点: 切块大小，重叠长度，语义完整性，检索精度
描述: 切块大小影响检索精度和上下文完整性。小块检索更精准但可能丢失上下文，大块保留更多语义但可能引入噪音。重叠长度用于保持跨块语义连贯性，需根据文档类型和业务场景调整。

题目: 选择嵌入模型（Embedding Model）的评估指标
关键点: MTEB，检索精度，推理速度，模型大小
描述: 评估Embedding模型好坏的指标包括：①MTEB等基准测试的检索精度；②推理速度和延迟；③模型大小和部署成本；④对领域数据的适配能力。

题目: 提升 RAG 检索质量的技术
关键点: 混合检索，重排（Rerank），查询改写，HyDE
描述: 提升检索质量的技术包括：①混合检索（结合关键词检索+向量检索）；②引入Rerank模型对初召结果重排；③查询改写/扩展优化用户输入；④HyDE（生成假设性答案用于检索）。

题目: "Lost in the Middle" 问题与缓解方法
关键点: 长文本，中间内容遗忘，上下文重排序
描述: “Lost in the Middle”指LLM在处理长上下文时，容易忽略中间位置的内容。缓解方法包括：①对检索结果按相关性重新排序，将高相关文档放在开头或结尾；②使用Context Window Re-weighting。

题目: RAG 系统部署的主要挑战
关键点: 延迟，数据更新，质量评估，规模化
描述: RAG系统实际部署的挑战包括：①检索延迟和系统响应速度；②知识库的动态增量更新（避免新旧文档分布不一致导致的检索偏差）；③检索和生成质量的持续评估；④大规模数据下的索引和检索性能。

题目: RAG 的常见问题与解决方案
关键点: 内容缺失，召回率，上下文拼接，格式错误
描述: 常见问题包括：①内容缺失（切片策略不合理/向量召回率低 → 调整分段长度、使用多向量检索）；②错过排名靠前的文档（优化向量距离计算或引入Rerank模型）；③脱离上下文/拼接语义边界丢失（采用基于语义的拼接）；④格式错误（统一索引格式、Prompt中约束输出模板）。

题目: 面向对象和面向过程的区别
关键点: 封装，继承，多态，数据与操作分离
描述: 面向过程以函数为核心，数据与操作分离，适合简单流程；面向对象以对象为核心，将数据与操作封装在一起，通过继承和多态提高代码复用性和扩展性。

题目: 接口与抽象类的区别
关键点: 多继承，构造方法，成员变量，访问修饰符，Java 8
描述: 接口支持多实现，不能有构造方法，成员变量默认为public static final，方法默认为public abstract（Java 8后可有default方法）；抽象类只能单继承，可有构造方法和各种成员变量。接口更强调"能做什么"，抽象类更强调"是什么"。

题目: JDK、JRE、JVM 三者的关系
关键点: 开发工具包，运行时环境，虚拟机
描述: JDK = JRE + 开发工具（如javac），JRE = JVM + 核心类库，JVM是Java程序运行的核心虚拟机。编写Java代码用JDK，运行Java程序只需JRE。

题目: Java 的基本数据类型与包装类
关键点: 8种基本类型，自动装箱，自动拆箱，缓存池
描述: 基本数据类型包括byte、short、int、long、float、double、char、boolean，对应包装类分别为Byte、Short、Integer、Long、Float、Double、Character、Boolean。自动装箱/拆箱是编译器在编译期自动插入的转换代码。Integer等包装类有缓存池（-128~127）。

题目: 什么是 Java 中的反射
关键点: 动态获取类信息，动态调用方法，框架基础
描述: 反射允许程序在运行时动态获取类的完整信息（字段、方法、构造器等），并动态创建对象和调用方法。Spring、MyBatis等框架底层大量使用反射实现依赖注入和ORM映射。

题目: Java 异常体系
关键点: Throwable，Error，Exception，运行时异常，检查异常
描述: Java异常根类是Throwable，分为Error（系统级错误，不可恢复）和Exception。Exception分为RuntimeException（运行时异常，非检查异常，如NullPointerException）和检查异常（需throws或try-catch，如IOException）。

题目: 什么是 Java 中的泛型
关键点: 类型参数，类型擦除，通配符，编译期检查
描述: 泛型提供了编译期类型安全检查，允许类、接口、方法在定义时使用类型参数。Java泛型通过类型擦除实现（编译后类型参数被替换为Object或边界类型），运行时无泛型信息。通配符（? extends T、? super T）用于解决泛型协变/逆变问题。

题目: Java 内存模型（JMM）
关键点: 主内存，工作内存，volatile，可见性，有序性
描述: JMM规定所有变量存储在主内存，每条线程有自己独立的工作内存。线程对变量的所有操作（读取、赋值）必须在工作内存中进行，不能直接读写主内存。volatile关键字保证可见性和一定程度的有序性，但不保证原子性。

题目: 类加载机制与双亲委派模型
关键点: 加载，验证，准备，解析，初始化，ClassLoader
描述: 类加载过程分为加载、验证、准备、解析、初始化五个阶段。双亲委派模型指类加载器收到加载请求时，先委托父类加载器加载，父类无法加载时再由自己加载，保证了核心类库的安全性（防止自定义类替换核心类）。

题目: String、StringBuffer、StringBuilder 的区别
关键点: 不可变，线程安全，性能
描述: String是不可变对象，每次修改都会创建新对象；StringBuffer是线程安全的可变字符序列（方法加synchronized）；StringBuilder是非线程安全的可变字符序列，性能最高。单线程环境下优先用StringBuilder，多线程环境下用StringBuffer。

题目: Java 中的锁机制
关键点: synchronized，ReentrantLock，读写锁，公平锁，非公平锁
描述: synchronized是JVM内置锁，使用简单但功能有限；ReentrantLock是API层面的锁，支持公平锁/非公平锁、可中断锁、超时锁等高级功能。ReadWriteLock适用于读多写少场景。synchronized在JDK 1.6后做了偏向锁、轻量级锁等优化，性能已经不差。

题目: 线程池的工作原理
关键点: 核心线程数，最大线程数，阻塞队列，拒绝策略
描述: 线程池的核心参数包括corePoolSize（核心线程数）、maximumPoolSize（最大线程数）、keepAliveTime（空闲存活时间）、workQueue（阻塞队列）、handler（拒绝策略）。任务提交时：核心线程未满则创建核心线程执行；核心线程满则入队；队列满则创建非核心线程；达到最大线程数则触发拒绝策略。

题目: 什么是 Java NIO
关键点: 非阻塞IO，Channel，Buffer，Selector
描述: NIO（New IO）提供非阻塞IO模型，核心组件包括Channel（通道）、Buffer（缓冲区）、Selector（选择器）。NIO允许单线程通过Selector管理多个Channel，适合高并发连接场景。

题目: 什么是设计模式？列举常见的设计模式
关键点: 单例，工厂，观察者，代理，模板方法
描述: 设计模式是面向对象设计中可复用的解决方案。常见的设计模式包括：单例模式（保证全局唯一实例）、工厂模式（创建对象）、观察者模式（一对多依赖）、代理模式（增强对象功能）、模板方法模式（定义算法骨架）、策略模式（算法可互换）等。

题目: Spring 中的 Bean 生命周期
关键点: 实例化，属性注入，初始化，销毁
描述: Spring Bean的生命周期包括：实例化（通过构造器创建对象）、属性注入（依赖注入）、Aware接口调用（BeanNameAware等）、BeanPostProcessor的前置处理、InitializingBean的afterPropertiesSet、自定义init方法、BeanPostProcessor的后置处理、使用阶段、销毁阶段（DisposableBean的destroy、自定义destroy方法）。

题目: Spring 事务传播行为
关键点: REQUIRED，REQUIRES_NEW，NESTED，MANDATORY，SUPPORTS
描述: Spring定义了7种事务传播行为：REQUIRED（默认，支持当前事务，无则新建）、REQUIRES_NEW（挂起当前事务，新建事务）、NESTED（嵌套事务，通过保存点实现回滚）、MANDATORY（必须已有事务）、SUPPORTS（有则加入，无则非事务）、NOT_SUPPORTED（非事务运行，挂起事务）、NEVER（非事务运行，有事务则报错）。

题目: MySQL 的存储引擎 InnoDB 和 MyISAM 的区别
关键点: 事务，外键，行锁，表锁，全文索引，崩溃恢复
描述: InnoDB支持事务（ACID）、行级锁、外键约束、崩溃恢复（通过redo log），是MySQL 5.5后的默认引擎；MyISAM不支持事务、只支持表级锁、支持全文索引，读性能好但写并发差。InnoDB适合高并发OLTP场景，MyISAM适合只读或读多写少场景。

题目: MySQL 的日志机制
关键点: binlog，redo log，undo log，两阶段提交
描述: MySQL三大日志：binlog（二进制日志，记录所有DDL/DML，用于主从复制和恢复，Server层）、redo log（重做日志，保证事务持久性，InnoDB层）、undo log（回滚日志，保证事务原子性和MVCC，InnoDB层）。两阶段提交（prepare + commit）保证binlog和redo log的一致性。

题目: MySQL 的 MVCC 机制
关键点: 多版本并发控制，Read View，undo log，隐藏列
描述: MVCC（Multi-Version Concurrency Control）通过保存数据的历史版本来实现非阻塞读。InnoDB为每行记录增加隐藏列（DB_TRX_ID事务ID、DB_ROLL_PTR回滚指针）。Read View决定事务能看到哪些版本的数据，RR级别下Read View在事务开始时创建，RC级别下每次查询都创建新的Read View。

题目: Redis 的数据结构与适用场景
关键点: String，Hash，List，Set，ZSet，Bitmap，HyperLogLog
描述: Redis支持多种数据结构：String（缓存字符串、计数器）、Hash（存储对象）、List（消息队列、最新列表）、Set（去重、交集并集）、ZSet（排行榜、延迟队列）、Bitmap（布隆过滤器）、HyperLogLog（基数统计）。

题目: Redis 的持久化机制
关键点: RDB，AOF，混合持久化
描述: Redis提供RDB（快照）和AOF（追加日志）两种持久化方式。RDB定期生成内存快照，适合备份恢复，但可能丢失最后一段时间的数据；AOF记录每条写命令，数据更完整但文件更大、恢复更慢。Redis 4.0后支持混合持久化（RDB + AOF），兼顾恢复速度和数据完整性。

题目: 缓存穿透、缓存击穿、缓存雪崩
关键点: 布隆过滤器，互斥锁，过期时间随机
描述: 缓存穿透（查询不存在的数据）→ 使用布隆过滤器或缓存空对象；缓存击穿（热点key过期）→ 使用互斥锁或"逻辑过期"；缓存雪崩（大量key同时过期）→ 设置随机过期时间、使用双缓存、服务降级。

题目: 分布式 CAP 理论
关键点: 一致性，可用性，分区容错性
描述: CAP定理指分布式系统最多只能同时满足一致性（Consistency）、可用性（Availability）、分区容错性（Partition Tolerance）中的两个。网络分区发生时，系统必须在C和A之间做权衡：CP系统（如ZooKeeper）保证一致性但可能不可用，AP系统（如Eureka）保证可用性但可能数据不一致。

题目: 什么是分布式事务？有哪些解决方案？
关键点: 两阶段提交，TCC，Saga，最大努力通知
描述: 分布式事务是跨多个服务或数据库的事务。常见解决方案：两阶段提交（2PC，强一致但性能差）、TCC（Try-Confirm-Cancel，业务侵入强）、Saga（长事务，通过补偿机制）、最大努力通知（最终一致性，适合异步场景）。

题目: 什么是微服务架构？与单体架构的区别
关键点: 服务拆分，独立部署，服务注册，服务发现，API网关
描述: 微服务将应用拆分为多个独立部署的服务，每个服务围绕业务能力构建，拥有独立数据库。相比单体架构，微服务提高了开发效率和系统弹性，但引入了服务发现、配置管理、链路追踪、分布式事务等复杂性。

题目: Agent 与传统对话系统的本质区别是什么
关键点: 动态推理，自主工具组合，长期记忆，自我修正
描述: 传统对话系统基于预设规则或有限状态机，决策能力有限；AI Agent具备动态推理生成解决方案的能力，可自主发现并组合工具，具备跨会话长期记忆管理和自我反思与修正机制。

题目: 什么是 ReAct 框架
关键点: Reasoning，Acting，思考-行动-观察循环
描述: ReAct = Reasoning + Acting，将"推理"（思考下一步该做什么）和"行动"（执行具体操作）交替进行。Agent先"思考"当前状态和目标，再"行动"调用工具或给出回答，通过"思考-行动-观察"的循环完成复杂任务。

题目: Agent 的规划能力有哪些实现方法
关键点: CoT，ToT，GoT，任务分解
描述: 赋予LLM规划能力的主流方法包括CoT（思维链，通过逐步推理解决问题）、ToT（思维树，探索多条推理路径）、GoT（思维图，允许思维之间相互连接）。

题目: 如何为 Agent 设计短期记忆和长期记忆系统
关键点: 滑动窗口，向量化存储，FAISS，记忆压缩
描述: 短期记忆可使用滑动窗口保存最近N轮对话，配合记忆压缩算法（如TF-IDF关键词提取）；长期记忆通过向量化存储将关键信息转为embeddings存入向量数据库（如FAISS、Milvus），通过相似性搜索实现检索。

题目: Tool Use（工具调用）的实现原理
关键点: Function Calling，工具注册表，参数校验，频率限制
描述: LLM通过Function Calling机制学会调用外部工具。设计上采用工具注册表模式（包含工具描述、API地址、频率限制等），调用时进行参数类型检查、权限白名单验证、调用频率限制，并设计异常处理流程（参数校验失败返回错误码，结果解析失败触发重试机制）。

题目: LangChain 和 LlamaIndex 的区别与选型
关键点: Agent编排，数据索引，应用场景
描述: LangChain侧重Agent编排和链式调用，适合构建复杂的多步骤Agent工作流；LlamaIndex侧重数据索引和检索，适合构建RAG系统。选型时根据核心需求决定：复杂Agent逻辑选LangChain，数据检索为主选LlamaIndex。

题目: 什么是多智能体系统
关键点: 多Agent协作，任务分解，通信协调
描述: 多智能体系统由多个LLM Agent协同工作，优势包括任务分解并行处理、角色分工专业化。引入的复杂性包括Agent间通信与协调成本增加、任务分配策略设计困难、一致性和冲突解决挑战。

题目: 如何确保 Agent 的行为安全可控
关键点: 对齐方法，安全护栏，人类反馈
描述: 确保Agent行为安全可控的方法包括：设计安全护栏（禁止某些操作）、人类反馈机制（human-in-the-loop）、RLHF对齐（让Agent行为符合人类价值观）、输出过滤和审核。

题目: 大规模 Agent 系统的资源调度策略
关键点: 多线程，多进程，GPU资源分配，时延与吞吐
描述: 大规模Agent系统在多线程/多进程场景下需要设计合理的资源调度策略，包括线程池管理、任务队列、优先级调度。在GPU资源有限条件下同时提供推理和微调服务时，需做资源分配和任务调度以平衡时延和吞吐。

题目: Agent 的 Prompt 如何设计和迭代
关键点: Prompt工程，意图补全，自动优化
描述: Agent的Prompt设计需要明确角色定位、工具使用规范、输出格式约束。当用户提出不完整请求时，需通过追问或上下文补全用户意图。可探索Prompt自动优化技术（如自动提示工程）。

题目: 什么是 A2A 框架
关键点: Agent间通信，协作协议
描述: A2A（Agent-to-Agent）是Agent之间的通信和协作框架，区别于普通Agent框架的核心在于它定义了Agent间标准化的通信协议，使不同Agent能够互操作和协同工作。

题目: 如何评估一个 Agent 系统的性能
关键点: 任务完成率，执行效率，工具调用准确性
描述: 评估Agent比评估基础LLM更复杂。评估维度包括任务完成率、执行效率、工具调用准确性、决策合理性等。需要构建专门的测试环境和基准来评估多步骤任务的完成情况。

题目: Agent 的容错与错误恢复机制
关键点: 故障容错，超时机制，feedback策略
描述: Agent在调用外部工具时需设计故障容错和超时机制。工具调用失败后需设计feedback策略，如重试、降级、人工介入等。

题目: Agent 中的记忆衰退与遗忘机制
关键点: 旧数据干扰，遗忘策略
描述: 为避免旧数据干扰新任务，Agent需设计记忆衰退机制（如基于时间衰减、重要性评分等），定期遗忘低频记忆，保持记忆系统的高效性。

题目: 如何让多个 Agent 协同工作
关键点: 协同机制，任务分配，冲突解决
描述: 多Agent协同需要设计具体的协同机制，如主从模式（主Agent分配任务，子Agent执行）、对等模式（Agent间协商）。需处理Agent误判导致的策略冲突。

题目: RAG 的完整工作流程
关键点: 数据接入，清洗，切片，向量化，索引存储，检索召回，过滤重排，上下文组装，生成
描述: 完整的RAG流水线包括：数据接入→文档清洗→文本切块→向量化（Embedding）→索引构建（存入向量数据库）→检索召回→过滤重排→上下文组装→LLM生成。

题目: RAG 中常见的"10大坑"及解决方案
关键点: 内容缺失，召回率，上下文脱离，格式错误，特异性错误
描述: 常见问题包括：①内容缺失（切片策略不合理/向量召回率低→调整分段长度、使用多向量检索）；②错过排名靠前的文档（优化向量距离计算或引入Rerank模型）；③脱离上下文（语义边界丢失→Context Window Re-weighting）；④格式错误（统一索引格式、Prompt约束输出模板）；⑤特异性错误（引入领域词向量或知识图谱增强）；⑥回答不全面（多通道检索：keyword + embedding）；⑦复杂PDF提取困难（布局识别+OCR+坐标级切分）。

题目: 如何选择合适的文本切块策略
关键点: 切块大小，重叠长度，语义完整性，检索精度
描述: 切块大小影响检索精度和上下文完整性。小块检索更精准但可能丢失上下文，大块保留更多语义但可能引入噪音。重叠长度用于保持跨块语义连贯性，需根据文档类型和业务场景调整。

题目: 如何选择嵌入模型（Embedding Model）
关键点: MTEB，检索精度，推理速度，模型大小
描述: 评估Embedding模型好坏的指标包括MTEB等基准测试的检索精度、推理速度和延迟、模型大小和部署成本、对领域数据的适配能力。

题目: 提升 RAG 检索质量的技术
关键点: 混合检索，Rerank，查询改写，HyDE
描述: 提升检索质量的技术包括：①混合检索（结合关键词检索BM25 + 向量检索）；②引入Rerank模型对初召结果重排；③查询改写/扩展优化用户输入；④HyDE（生成假设性答案用于检索）。

题目: "Lost in the Middle"问题
关键点: 长文本，中间内容遗忘，上下文重排序
描述: "Lost in the Middle"指LLM在处理长上下文时，容易忽略中间位置的内容。缓解方法包括对检索结果按相关性重新排序，将高相关文档放在开头或结尾，以及使用Context Window Re-weighting。

题目: RAG vs SFT 的核心区别
关键点: 知识注入方式，更新成本，幻觉，可溯源
描述: SFT是在模型内部"灌知识"（靠记忆），RAG是让模型"查资料"（靠检索）。RAG的优势：知识更新快、成本低、回答可溯源；SFT的优势：推理更自然流畅。RAG主要解决大模型知识过时、缺少私有数据、回答不可溯源等问题。

题目: 评估 RAG 系统的指标体系
关键点: 检索质量，生成质量，Hit Rate，MRR，Faithfulness
描述: 评估RAG系统需从检索和生成两个阶段分别评估。检索阶段指标包括Hit Rate（命中率）、MRR（平均倒数排名）、Recall；生成阶段指标包括Faithfulness（忠实度，回答是否基于检索内容）、Answer Relevance（答案相关性）。

题目: 高级 RAG 范式（自适应检索）
关键点: 多次检索，生成中检索，Agentic RAG
描述: 传统的RAG是"先检索后生成"。高级范式包括在生成过程中进行多次检索（Iterative RAG）或自适应检索（Adaptive RAG），让Agent根据生成需要动态决定何时检索、检索什么。

题目: RAG 系统部署的挑战
关键点: 延迟，数据更新，质量评估，规模化
描述: RAG系统实际部署的挑战包括：①检索延迟和系统响应速度；②知识库的动态增量更新（避免新旧文档分布不一致导致的检索偏差）；③检索和生成质量的持续评估；④大规模数据下的索引和检索性能。

题目: 什么是 Graph RAG
关键点: 知识图谱，图数据库，关系检索
描述: Graph RAG使用图数据库或知识图谱来增强或替代传统的向量数据库检索。适合需要处理实体间复杂关系的场景（如多跳问答、关系推理），通过图谱结构捕捉实体间的语义关联。

题目: RAG 中的召回（Recall）是什么
关键点: 向量召回，BM25，Hybrid Search，Query Rewrite
描述: RAG中的召回指从知识库中检索相关文档的过程。生产级RAG不只使用向量召回，还会结合BM25（稀疏检索）进行Hybrid Search（混合检索）。Query Rewrite和HyDE分别解决查询表述不准确和查询与文档语义gap的问题。

题目: RAG-Fusion 的工作原理
关键点: 多查询融合，结果合并，排序优化
描述: RAG-Fusion通过生成多个相关查询，分别检索后将结果融合排序。核心思想是利用LLM生成查询的变体，扩大召回覆盖面，再通过RRF（倒数排名融合）等算法合并结果。

题目: RAG 中的意图识别与子问题分解
关键点: NLU，查询分解，多步检索
描述: 在RAG系统中，意图识别用于理解用户真正想查什么。对于复杂查询，可进行子问题分解——将复杂问题拆解为多个子问题分别检索，再汇总答案。

题目: RAG 系统中的 Function Calling 与 MCP
关键点: 工具调用，模型上下文协议
描述: 在RAG系统中，Function Calling允许Agent在检索后调用外部工具（如数据库查询、API调用）获取额外信息。MCP（Model Context Protocol）是标准化模型与工具之间交互的协议，使Agent能更规范地调用外部能力。
